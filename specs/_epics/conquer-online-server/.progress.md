## Learnings
- M1 scope is deliberately narrow: auth + world entry + movement only. Combat/AI/items deferred.
- Two .NET processes (AccountServer + GameServer) share no in-process memory — one-time token handoff needs a shared store (MySQL or Redis); decision is an open question before FR-8 implementation.
- DMAP passability parsing is required for M1 movement validation; Twin City map (ID 1002) is the minimum required map.
- GameServer packet encryption scope (RC5 or none) is unresolved and must be confirmed against client 5017 protocol captures before implementing the packet layer.
- Account registration packet is out of scope for M1; test accounts seeded via Docker init SQL.
- Screen-range broadcast uses linear scan for M1 (≤10 concurrent); data structure should be designed to allow future spatial indexing without full rewrite.
- Map-partitioned `ConcurrentDictionary<mapId, ConcurrentDictionary<charId, GameSession>>` satisfies both M1 linear scan and future spatial index without refactoring callers.
- Token consume uses `UPDATE … WHERE consumed_at IS NULL AND expires_at > NOW()` — `rows affected == 1` is the atomic success signal; no additional locking required.
- `ICipher` abstraction (`Encrypt/Decrypt` on `Span<byte>`) lets GameServer ship with `NullCipher` and swap to a real cipher with zero structural refactor once 5017 GS encryption is confirmed.
- Startup migration approach: inline idempotent SQL (`CREATE TABLE IF NOT EXISTS`) executed via Dapper at process start — no FluentMigrator or EF dependency for M1.
- Graceful shutdown via `IHost` `CancellationToken` triggers a final position save for all active sessions before exit.
